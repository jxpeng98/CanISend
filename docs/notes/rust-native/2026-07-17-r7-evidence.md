# R7.3 profile evidence implementation note

## Accepted scope

R7.3 completes the reusable profile-evidence branch of the Rust-native workflow. It does not yet match evidence to
job criteria or make an apply/hold/skip decision; those remain R7.4 and R7.5.

## Revisioned source intake

`profile source add` imports bounded, regular, non-symlink Markdown, plain-text, or JSON files. The original bytes and
deterministically normalized UTF-8 text are separate immutable artifacts. Public command responses return source and
artifact metadata without embedding the private body. A workspace-level profile revision advances on every import.

SQLite migrations 6 and 7 add profile source identities/revisions, task and stage profile-revision pins, candidate
hashes, source-bound evidence identities, and catalog-to-item revision provenance. Profile changes invalidate only
evidence and its descendants; the independent job parse and criteria branch remains current.

## Agent proposal boundary

`task prepare --operation evidence-normalize` freezes the current profile revision and every current normalized
profile artifact revision/hash. Host-agent and configured-provider modes share the same schema-first, semantic, and
exact byte-span validators. Configured-provider export requires both private-read and provider-send consent.

The agent returns `canisend.evidence-proposals/v2`, which deliberately has no catalog ID, evidence ID, confirmation,
exclusion, or revision fields. It may only propose kind, factual summary, exact quote/span, and sensitivity. The core
rejects out-of-scope or changed sources and generates an unconfirmed EvidenceCatalog plus stable UUIDv7 IDs only
after validation. Candidate-hash persistence makes a repeated identical completion idempotent and rejects a different
candidate after commit.

## User decision and revisions

The public decision loop is:

```text
canisend profile evidence proposed --job JOB_ID --json
canisend profile evidence export --job JOB_ID --destination evidence.json --json
canisend profile evidence confirm --job JOB_ID --file evidence.json --json
canisend profile evidence show --job JOB_ID --json
```

Export marks the proposal ready for explicit review. The user may correct kind, summary, quote/span, sensitivity, and
`excluded`, but cannot add, remove, or replace core-generated IDs. Confirmation revalidates the current profile
revision, source scope, artifact hashes, and exact UTF-8 bytes before one transaction commits the final catalog,
evidence item revisions, dependencies, audit event, and stage state.

Exporting an already confirmed catalog creates the supported revision template. A subsequent confirmation keeps the
catalog, artifact, and evidence identities stable while the core advances both catalog and item revisions. Any such
revision invalidates match and later outputs without disturbing parse or criteria.

## Embedded host contract

The capability and stage registries now truthfully expose `evidence.lifecycle` and the evidence stage as available.
Codex, Claude, and generic packs include the evidence-normalize prompt plus EvidenceProposal, EvidenceCatalog, and
Evidence schemas. The guides explicitly prohibit agents from assigning IDs or confirming evidence for the user.
