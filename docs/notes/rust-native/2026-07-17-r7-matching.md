# R7.4 criterion-to-evidence matching implementation note

## Accepted scope

R7.4 completes the derived matching layer between user-confirmed job criteria and user-confirmed profile evidence.
It does not make the user's apply/hold/skip decision or choose application documents; those remain R7.5.

## Proposal and output contracts

The agent candidate is `canisend.evidence-match-proposals/v2`. It repeats the exact Criteria and EvidenceCatalog
artifact references and supplies exactly one proposal per criterion. A proposal contains:

- the exact criterion identity and revision;
- zero or more exact evidence identities and revisions;
- `strong`, `partial`, `gap`, or `unknown` support;
- a factual rationale;
- an explicit unresolved gap where the strength requires one;
- a bounded list of claims that downstream drafting must not make.

The proposal contract has no match identity or revision fields. After all references pass validation, core generates
stable UUIDv7 match identities and a final `canisend.evidence-matches/v2` MatchSet. This follows the same authority
rule as evidence normalization: agents propose bounded reasoning, while CanISend owns identity and durable state.

Strong matches require cited evidence and forbid a remaining gap. Partial matches require both cited evidence and a
non-empty gap. Gap and unknown matches require a non-empty gap and cannot claim supporting evidence. Evidence and
prohibited-claim lists are bounded and reject duplicate identities or duplicate claims.

## Revision-bound task transaction

`task prepare --operation evidence-match` is available only when both criteria and evidence stages are complete. It
freezes the current job and profile revisions plus the exact Criteria and EvidenceCatalog artifact revisions/hashes.
Host-agent and configured-provider modes share one validator and one atomic commit path; provider mode adds explicit
send consent.

Completion loads the two declared artifacts from verified blobs and rejects:

- a different artifact identity, revision, or hash;
- an unknown or wrong-revision criterion;
- missing or duplicate criterion coverage;
- unknown or wrong-revision evidence;
- unconfirmed or user-excluded evidence;
- strength/gap/evidence combinations that violate the semantic contract.

The resulting EvidenceMatches artifact depends on both exact input artifacts. Completion atomically records the task
result and audit event, completes the match stage, and makes the plan stage ready. Repeating the identical candidate
is idempotent; a different candidate after commit conflicts.

## Freshness and host use

Criteria rerun/confirmation and evidence correction/revision already invalidate match and all descendants. Prepared
match tasks become stale, old MatchSet artifacts are marked stale, and the match stage becomes ready only after both
new upstream artifacts are complete. The integration test explicitly attempts to cite excluded evidence, revises the
catalog, observes the old task become stale, and then recomputes against evidence revision 2.

`match show --job JOB_ID --json` returns only the current validated MatchSet. Capability and stage registries now mark
`match.lifecycle` and the match stage available. Codex, Claude, and generic packs include the match prompt and both
proposal/output schemas, and their guides prohibit invented IDs and revision-free citations.
