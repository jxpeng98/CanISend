# R7.5 application decision and document-plan implementation note

## Accepted scope

R7.5 closes the Rust-native intake-to-plan workflow. It consumes only the current validated EvidenceMatches artifact;
it does not reopen raw job or profile bodies. The user, not an agent or heuristic, owns the final `apply`, `hold`, or
`skip` decision.

## Candidate and durable contracts

`canisend.application-plan-candidate/v2` is the editable decision boundary. It contains the exact current match
artifact reference, an application strategy, all four supported document kinds, and blockers derived by core. It
contains no plan or document IDs. `canisend.application-plan/v2` is the committed output; CanISend assigns stable
UUIDv7 identities and revisions to the plan and every document entry.

Strategy requires a positioning statement and at least one bounded priority. Each Cover Letter, CV, Research
Statement, and Teaching Statement is classified as required, optional, or omitted. Included documents must select a
host-agent or configured-provider executor; omitted documents cannot have one. `skip` requires all documents to be
omitted.

The export default is `hold`. Re-exporting a confirmed plan preserves the user's existing strategy and document
choices while refreshing the exact match reference and core-derived blockers. Reconfirmation retains plan/document
IDs and increments their revisions.

## Derived blocker and decision gate

CanISend joins every match to the exact criterion revision. Non-strong essential criteria produce blocking plan
records; non-strong desirable criteria produce warnings; informational criteria do not block. Agents and users
cannot remove, add, or rewrite these records: `plan confirm` recomputes and requires exact equality inside the same
immediate SQLite transaction.

An `apply` decision is rejected while any blocking record remains. `hold` and `skip` may be committed so the workflow
can preserve a real decision without pretending that drafting is authorized. Draft becomes ready only when the
current plan head exactly matches the Plan stage output, the decision is `apply`, and the blocking count is zero.
Body-free workflow context reports a decision or blocker code and count, never source text or plan prose.

## Storage and invalidation

SQLite migration 8 adds `application_plan_heads`, a narrow projection containing the workflow run, exact plan
artifact revision, decision, blocking count, and update time. The full plan remains an immutable content-addressed
artifact depending on the exact EvidenceMatches revision.

Plan revisions stale and clear Draft/Review/Package/Render outputs and prepared tasks. Match, criteria, evidence,
profile, or job changes already invalidate the plan transitively. A stale or mismatched plan head cannot open Draft.
Generic `workflow complete --stage plan` is rejected so callers cannot bypass `PlanService` validation.

## CLI and host workflow

The available `plan.lifecycle` surface is:

```text
canisend plan export --job JOB_ID --destination application-plan.json --json
canisend plan confirm --job JOB_ID --file application-plan.json --json
canisend plan show --job JOB_ID --json
```

Workflow next actions route a ready Plan stage to `plan export`. Codex, Claude, and generic host packs instruct the
host to explain blockers and the safe hold default, obtain the user's explicit decision, and avoid treating readiness
as permission to submit an application.
