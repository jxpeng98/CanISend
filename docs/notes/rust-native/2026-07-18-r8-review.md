# R8.3 revision-bound review note

## Outcome

R8.3 makes Review an available workflow stage. A `document.review` task freezes the exact current DocumentSet, all
member document artifacts, ApplicationPlan, EvidenceMatches, Criteria, and EvidenceCatalog. The same candidate and
commit boundary supports host-agent and configured-provider execution; provider mode retains separate private-read
and provider-send consent.

The task returns an ID-free `canisend.review-candidate/v2`. CanISend validates every document-set, document, section,
claim, and placeholder target against the exact declared inputs before it assigns finding identities.

## Two review authorities

ReviewFindings separates two authorities:

- `deterministic`: findings produced by Rust checks and not dismissible by an agent or user;
- `human-review`: semantic findings proposed by the bounded reviewer and subject to explicit user disposition.

The deterministic pass independently checks:

- evidence and criterion citations against the current match/evidence/criteria chain;
- unresolved required placeholders;
- sections with prose but no declared claims;
- literal prohibited-claim phrases after conservative normalization;
- identical repeated claim text whose classification or citation signature differs across documents.

The bundled review prompt asks the host to inspect semantic support, paraphrased prohibited claims, broader
cross-document inconsistencies, tone, motivation, emphasis, and user judgement. It explicitly avoids duplicating the
mechanical pass.

## Dispositions and revisions

All new findings begin `open`. Human-review findings may be set to `accepted-risk` or `dismissed` only through
`review export` followed by explicit user editing and `review confirm`. Each selected disposition requires the exact
finding revision and a rationale; CanISend records the user and timestamp. A changed disposition increments both the
finding revision and the ReviewFindings artifact revision while preserving their IDs and exact upstream dependencies.

Deterministic findings cannot receive either disposition. They remain open until a new current document set is
drafted and reviewed. This keeps a user-facing acknowledgement from masquerading as a passed deterministic check.

## Persistence and invalidation

SQLite migration 10 adds `review_heads`, binding the current review revision to the exact DocumentSet and storing
body-free deterministic-blocker and pending-human counts for R8.4. Review completion unlocks Package; a later user
disposition invalidates Package/Render descendants without rerunning Review. Any upstream plan, evidence, profile,
job, draft, or scoped workflow change stales the review artifact and clears its head.

## Agent and CLI surface

- `task prepare --operation document-review`;
- `review show`;
- `review export`;
- `review confirm`.

`review.lifecycle` and the Review stage are now available. Codex, Claude, and generic host packs include the review
prompt plus review candidate, findings, and disposition schemas; each pack contains 25 integrity-manifested files.

The R8.3 quality gate passed:

- 65 Rust tests;
- Clippy across all targets/features with warnings denied;

- 44 embedded-resource checks;
- release compilation and packaged host-agent smoke.

The integration path verifies a configured-provider review with nine exact inputs, rejection of an invented target,

an accepted-risk revision, Package unlock, and full review invalidation after a profile change.

## Next boundary

R8.4 consumes only current DocumentSet and ReviewFindings heads. It must produce body-free readiness reasons, block
on open deterministic blockers or pending human decisions, reject mixed upstream revisions, verify every required
planned document, and preserve the invariant that readiness never means submission.
