# Post-0.7 CanISend Roadmap Draft

**Status:** Draft; it cannot become the published next roadmap until the Stable source gate consumes an RC feedback
snapshot.

**Evidence authority:** [`release/feedback-snapshot.json`](../../../release/feedback-snapshot.json)

## Measured baseline

The 2026-07-18 Alpha snapshot contains **No public user issue**: zero open, zero closed, and zero total GitHub issues.
Every one of the 12 public release assets showed one download, including all five native archives. Those counts include
the maintainer verification performed during Alpha qualification, so they are not unique-user, adoption, retention,
or platform-demand measurements. CanISend has no default telemetry and this roadmap will not infer private use from
workspace data.

The release process did produce two reproducible engineering findings:

1. a Windows native/Git Bash temporary-directory path mismatch in the clean documented-workflow matrix; and
2. CRLF checkout conversion causing a false Beta contract-freeze drift on Windows.

Both were fixed without weakening the binary, contract, or release gates. This is evidence for continuing
cross-platform source/packaged-boundary testing; it is not evidence that users requested a new product feature.

## Beta/RC refresh gate

Before Stable, replace the Alpha baseline with public metadata captured after the latest RC:

- count open and closed issues since Beta, retaining issue numbers but no copied private bodies;
- classify verified reports by data loss, security/privacy, protocol/workspace compatibility, rendering,
  installation, discovery source, agent-host, and documentation;
- record public Beta/RC asset downloads while continuing to state that download counts are not unique users;
- link every release qualification defect and its resolution or open blocker;
- publish this roadmap only after every blocker class is clear and priorities cite the refreshed snapshot.

Stable must fail if the snapshot stage is not `rc` or this roadmap still says `draft`.

## Candidate outcomes after 0.7

These are hypotheses, not committed features. The RC refresh decides whether they remain, move, or disappear.

### O1 — Release portability and supported installation channels

Keep cross-platform checkout, path, signing, package-manager install/upgrade/uninstall, and archive-boundary regression
coverage. This is the only candidate directly supported by current qualification findings.

### O2 — Expand job discovery sources by measured demand

Add institution boards, discipline-specific feeds, or API-backed sources only when public issues identify repeated
sources and provide safe reproducible examples. Preserve supplied URL, PDF, JSON, CSV, RSS/Atom, and manual intake as
first-class paths rather than requiring an API.

### O3 — Improve Codex and Claude workflow ergonomics

Use public reports about task preparation, candidate completion, host-pack refresh, and error remediation to reduce
friction while retaining one Agent v2 validation boundary. Do not fork product state by host.

### O4 — Evaluate scanned-PDF OCR as an optional boundary

Proceed only if Beta/RC feedback shows repeated scanned-advert failures. Any design must preserve offline/native
installation, explicit resource licensing, bounded processing, and a clear distinction between extracted and inferred
text.

### O5 — Evaluate Linux arm64

Proceed only if public demand and a trustworthy native or reproducible runner justify adding a sixth signed/provenance
qualified archive. Do not claim support from cross-compilation alone.

## Prioritization rule

Security, privacy, data-loss, and workspace-corruption blockers always precede feature work. After blockers, rank an
outcome by reproduced public reports, affected supported targets, workflow impact, implementation risk, and the
strength of release evidence. A zero-report baseline cannot justify expanding scope by itself.
