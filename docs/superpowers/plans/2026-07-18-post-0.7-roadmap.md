## Candidate outcomes after 0.7

These are hypotheses, not committed features. The RC refresh decides whether they remain, move, or disappear.

### O1 — Release portability and supported installation channels

Keep cross-platform checkout, path, signing, package-manager install/upgrade/uninstall, and archive-boundary regression
coverage. This is the only candidate directly supported by current qualification findings.

### O2 — Expand job discovery sources by measured demand

Add institution boards, discipline-specific feeds, or API-backed sources only when public issues identify repeated
sources and provide safe reproducible examples. Preserve supplied URL, PDF, JSON, CSV, RSS/Atom, and manual intake as
first-class paths rather than requiring an API.

### O3 — Improve Codex and Claude workflow ergonomics

Use public reports about task preparation, candidate completion, host-pack refresh, and error remediation to reduce
friction while retaining one Agent v2 validation boundary. Do not fork product state by host.

### O4 — Evaluate scanned-PDF OCR as an optional boundary

Proceed only if Beta/RC feedback shows repeated scanned-advert failures. Any design must preserve offline/native
installation, explicit resource licensing, bounded processing, and a clear distinction between extracted and inferred
text.

### O5 — Evaluate Linux arm64

Proceed only if public demand and a trustworthy native or reproducible runner justify adding a sixth signed/provenance
qualified archive. Do not claim support from cross-compilation alone.

## Prioritization rule

Security, privacy, data-loss, and workspace-corruption blockers always precede feature work. After blockers, rank an
outcome by reproduced public reports, affected supported targets, workflow impact, implementation risk, and the
strength of release evidence. A zero-report baseline cannot justify expanding scope by itself.
