## Approved product direction after 0.7

### G1 — Native desktop GUI over the shared Rust application core

The product owner has approved a simple local desktop interface as the first committed `0.8` extension. This is a
strategic product direction, not an inference from the zero-report Alpha baseline. Its justification is to make the
existing native workflow usable without a terminal while preserving CLI and Agent v2 as first-class interfaces.

The [native desktop GUI roadmap](2026-07-19-native-desktop-gui-roadmap.md) fixes the boundaries:

- implementation begins only after `0.7` Stable, so the R11 feature freeze is unchanged;
- GUI and CLI call one typed Rust application facade and one authoritative workspace;
- all ordinary state-changing CLI operations receive bounded GUI actions, not arbitrary shell execution;
- the GUI manages a safe registry of local workspaces without treating Git or cloud sync as a requirement;
- supplied URLs, PDFs, JSON, CSV, RSS/Atom, manual files, and agent paths remain first-class;
- the first GUI Alpha targets macOS arm64/Intel, Windows x64, and Linux x64 glibc while musl remains CLI-only; and
- packaging, accessibility, privacy, rollback, uninstall, and workspace retention are release gates.

Verified public reports still decide the ordering of fixes within G1 and whether later visual or platform expansion
is justified. Security, privacy, data-loss, workspace-corruption, and current Stable release blockers precede GUI
feature work.

## Candidate outcomes after 0.7

These are hypotheses, not committed features. The RC refresh decides whether they remain, move, or disappear.

### O1 — Release portability and supported installation channels

Keep cross-platform checkout, path, signing, package-manager install/upgrade/uninstall, and archive-boundary regression
coverage. This is the only candidate directly supported by current qualification findings.

### O2 — Expand job discovery sources by measured demand

Add institution boards, discipline-specific feeds, or API-backed sources only when public issues identify repeated
sources and provide safe reproducible examples. Preserve supplied URL, PDF, JSON, CSV, RSS/Atom, and manual intake as
first-class paths rather than requiring an API.

### O3 — Improve Codex and Claude workflow ergonomics

Use public reports about task preparation, candidate completion, host-pack refresh, and error remediation to reduce
friction while retaining one Agent v2 validation boundary. Do not fork product state by host.

### O4 — Evaluate scanned-PDF OCR as an optional boundary

Proceed only if Beta/RC feedback shows repeated scanned-advert failures. Any design must preserve offline/native
installation, explicit resource licensing, bounded processing, and a clear distinction between extracted and inferred
text.

### O5 — Evaluate Linux arm64

Proceed only if public demand and a trustworthy native or reproducible runner justify adding a sixth signed/provenance
qualified archive. Do not claim support from cross-compilation alone.

## Prioritization rule

Security, privacy, data-loss, and workspace-corruption blockers always precede feature work. G1 begins only after
`0.7` Stable and follows its dedicated phase gates. After blockers, rank the remaining candidate outcomes by
reproduced public reports, affected supported targets, workflow impact, implementation risk, and the strength of
release evidence. A zero-report baseline cannot justify expanding candidate scope by itself.
